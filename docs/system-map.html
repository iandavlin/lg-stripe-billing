4. Flow walkthroughs

A. New subscription (Stripe Checkout → webhook → WP role)

  1. Frontend POST /v1/checkout with price_id email country.
  2. CheckoutService asks ProductRepository to resolve price: resolvePriceForCountry may swap to dev/region price.
  3. CheckoutService calls StripeGateway.createCheckoutSession, returns clientSecret.
  4. User completes payment in Stripe's embedded form. Stripe redirects to /v1/return.
  5. Stripe asynchronously POSTs to /v1/webhook: checkout.session.completed.
  6. WebhookHandler: signature check → idempotency check → dispatch to onCheckoutCompleted.
  7. Retrieve the subscription from Stripe (for the price ID). Look up tier via ProductRepository.tierForPrice.
  8. CustomerManager.findOrCreate → row in customers.
  9. SubscriptionRepository.upsert → row in subscriptions.
  10. EntitlementManager.grantMembershipFromSubscription → row in entitlements (kind=membership_tier, source=subscription).
  11. WpRoleSync.sync(customerId) → cross-DB UPDATE of wp_usermeta.wp_capabilities.
  12. Return 200 to Stripe.

Tables touched: customers · subscriptions · entitlements · processed_events · audit_log · wp_user_bridge · looth_dev.wp_usermeta

B. Tier upgrade (monthly looth2 → monthly looth3)

  1. User changes plan in Stripe Customer Portal.
  2. Stripe fires customer.subscription.updated with new price.
  3. WebhookHandler.onSubscriptionUpdated: looks up customer by stripe_customer_id.
  4. SubscriptionRepository.upsert updates the same row in-place (UPSERT keyed on stripe_subscription_id).
  5. EntitlementManager.grantMembershipFromSubscription: revokes prior membership_tier entitlement sourced from this subscription, grants new one.
  6. WpRoleSync.sync overwrites wp_capabilities — removes looth2, adds looth3, preserves everything else.

Tables touched: subscriptions · entitlements (2 rows affected) · audit_log · wp_usermeta

C. Cancellation (sub ends at period end)

  1. User clicks "cancel" in portal. Stripe fires customer.subscription.updated with cancel_at_period_end=true.
  2. onSubscriptionUpdated upserts sub (flag stored), no role change yet — user still has paid-for access.
  3. Period ends. Stripe fires customer.subscription.deleted.
  4. onSubscriptionDeleted: upsert with status='canceled', canceled_at=NOW.
  5. EntitlementManager.revokeForSubscription → entitlements.revoked_at = NOW().
  6. WpRoleSync.sync: no active tier → writes looth1 (lapsed), preserves admin/bbp roles.

D. Refund or chargeback (planned)

  1. Admin refunds via Stripe dashboard. Stripe fires charge.refunded (or charge.dispute.created).
  2. WebhookHandler.onRefund: find the related order/subscription by charge metadata.
  3. EntitlementManager.revokeForSubscription or .revokeForOrder (to be added).
  4. Write to audit_log with action='refund_revoke' for visibility.
  5. WpRoleSync.sync → user drops to looth1.

Not yet implemented. Add as part of adapters pass.

E. Reconciler tick (cron, hourly)

  1. Cron invokes bin/reconcile.
  2. Reconciler.run → WpRoleSync.syncAll.
  3. For each row in wp_user_bridge: compute desired tier from active entitlements, compare against current wp_capabilities, UPDATE if drift found.
  4. Returns SyncSummary: checked / updated / unchanged / errors.
  5. Summary written to log + (optionally) emailed to admin.

Safety net. Catches any webhook that silently failed mid-sync, plus one-time memberships whose expires_at passed with no Stripe event.
